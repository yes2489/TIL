# 소프트웨어 개발 방법론 (Software Development Methodology)
> 소프트웨어를 개발할 때 **어떤 관점·철학·원칙으로 개발을 수행할 것인지**를 정의한 방법   

*생명주기 모델 = 단계의 흐름 | 개발 방법론 = 개발을 바라보는 방식*

## 구조적 개발 방법론 (Structured Development)
> 전체 시스템을 **기능(Function)** 단위로 나누어 개발하고, 이를 통합하는 **분할과 정복** 방식의 방법론    
- 상위 기능에서 하위 기능으로 점진적으로 세분화하는 **하향식(Top-down) 접근**을 사용함  
- 각 기능의 **입력–처리–출력 흐름**을 명확히 하는 데 초점을 둠  
- 절차와 문서가 명확하나, **변경과 확장에 유연성이 낮음**


## 정보공학 개발 방법론 (Information Engineering Development)
> 시스템을 기능보다 **데이터 중심**으로 바라보고, 핵심 정보 구조를 먼저 정의하는 방법론   
- 데이터 간 관계를 시스템의 뼈대로 삼아 그 위에 기능을 구현함  
- 데이터의 **일관성과 통합**을 중시하여 대규모 정보 시스템에 적합함  
- 초기 분석 부담이 크고 **변화 대응은 느린 편**임

## 객체지향 개발 방법론 (Object-Oriented Development)
> 현실 세계의 개념을 **객체**로 추상화하여 시스템을 구성하는 방법론  

- 데이터와 행위를 객체로 묶어 **캡슐화**하고, 객체 간 상호작용으로 시스템을 표현함  
- **상속과 다형성**을 통해 재사용성과 확장성을 확보함  
- 초기 설계 품질에 따라 **유지보수성과 확장성의 차이가 큼**

## 컴포넌트 기반 개발 방법론 (CBD: Component Based Development)
> 시스템을 **재사용 가능한 컴포넌트**의 집합으로 보고 개발하는 방법론  
- 컴포넌트를 조립·결합하여 시스템을 구성함으로써 생산성과 품질을 높이는 데 초점을 둠  
- 컴포넌트 간 **인터페이스의 명확성**과 독립성을 중요하게 다룸  
- 재사용이 가능한 환경에서는 개발 효율이 높으나, **적절한 컴포넌트 확보가 전제 조건**임

## 애자일 개발 방법론 (Agile Development)
> 계획보다 **변화 대응**과 **고객 협업**을 중시하는 방법론  
- 짧은 개발 주기를 반복하며 **실행 가능한 결과물**을 지속적으로 제공함  
- 문서보다 **동작하는 소프트웨어**를 중심으로 개발을 진행함  
- 요구사항 변화가 잦은 환경에 적합하나, **대규모 관리에는 추가적인 통제가 필요함**
